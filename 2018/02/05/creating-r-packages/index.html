Why create an R package?

For most of us R users, creating a package was not a real consideration. It felt like creating and distributing a package was the domain of the super users. Things have changed and now there are good reasons for the typical R user to learn about and create packages on their own. Things have also changed quite a bit in terms of the level of difficulty of creating packages –packages like devtools and roxygen2 will do a good chunk of the heavy lifting for us. There is a bit added complication to release a package on CRAN, but making a package available almost couldn’t be simpler. The difficulty has come down quite a bit, but why would the typical R user want to create a package. Clearly a package is the primary medium for sharing robust coding projects aimed at the general R community. But what about code for yourself or a smaller community? What about sharing data and datasets? These both can be useful for one’s everyday workflow, but can also be used in combination with an academic paper to ensure that the results are reproducible. Upshot: There are many advantages to creating your own package even if it never leaves your machine. Seeing how easy it is to create a package will make the advantages to bundling your code and/ or data into a package well worth the time.

In what follows I will outline the steps of creating a basic package with a single function and an included dataset. I will cover creating the scaffolding/ directory structure, where to add code and data, how to document your package so that you get all the bells and whistles of the help functions in R and RStudio, how to create a vignette to provide examples of the functionality of the package and finally how to prepare it for release on GitHub.

Lay the groundwork

Although your functions and data are the content of a package, there is a basic set of directories and files that need to be created to set the stage for the package to work. To get started, fire up RStudio and make sure you have the devtools and roxygen2 packages installed.

install.packages(c("devtools", "roxygen2"))

The next step is to create the scaffolding for the package. Using the RStudio GUI interface, create an RStudio Project. Choose ‘New Directory’ and then select ‘R Package’ from the project types. This will bring you to a dialogue box where you will name your package, I’ve named mine packager, and choose the parent directory where you want it to live. By default the R Package project will have the option to create a git repository checked. Unless you do not want to git to share your project on GitHub, leave this option as is. If this is your first time working with git, I suggest this guide to get you started with the basics.

Package names have certain requirements. They cannot contain numerals or symbols. In most cases it is best to use a name that captures the main idea of your package and is in all lowercase. Camel case is acceptable, but that’s a matter of preference.

Accept this configuration and RStudio will open with the basic scaffolding for your package.

├── DESCRIPTION
├── NAMESPACE
├── R
│   └── hello.R
├── man
│   └── hello.Rd
└── packager.Rproj

This starter template includes two directories R/ and man/. R/ is where you will add your functions and man/ is where the documentation for these function will be stored. To get you started, an example function hello() is automatically generated –the function itself and its documentation. The other two files are the DESCRIPTION and NAMESPACE files. We can leave the NAMESPACE file as is, but we will need to edit the DESCRIPTION file to personalize our package.

Personalize the package

Let’s go ahead and start personalizing our package. Open the DESCRIPTION file.

Package: packager
Type: Package
Title: What the Package Does (Title Case)
Version: 0.1.0
Author: Who wrote it
Maintainer: The package maintainer <yourself@somewhere.net>
Description: More about what it does (maybe more than one line)
    Use four spaces when indenting paragraphs within the Description.
License: What license is it under?
Encoding: UTF-8
LazyData: true

Edit this file replacing the boilerplate content with the details describing your package.
